# FAQ

## Что делает GDELib

GDELib сохраняет и читает последовательность ячеек в собственном бинарном формате. Пользователь работает через `DEObject`: добавляет значения, сохраняет их в файл, потом снова открывает файл и получает значения обратно.

## Когда её использовать

Когда вам нужен локальный прикладной формат хранения:

- для save/load сценариев;
- для desktop-приложений и игр;
- для утилит, которым не нужна база данных;
- когда важна компактная API-обвязка вокруг собственного формата.

Если вам нужен общий обменный формат между системами, полноценное шифрование или конкурентный доступ, GDELib — не лучший выбор.

## Чем отличается NuGet от прямого подключения DLL

### NuGet

- проще обновлять версию;
- проще воспроизводить сборку;
- проще подключать пакет в CI и новых проектах.

### Прямое подключение DLL

- полезно в офлайн-средах;
- подходит для legacy-решений;
- требует вручную выбрать правильную сборку из `lib/net6.0`, `lib/netstandard2.1` или `lib/net472`. `[confirmed by package metadata]`

## С чего начать

Минимальный путь такой:

1. установить пакет `GDELib`;
2. создать папку хранения;
3. создать `DEObject`;
4. добавить несколько значений через `CreateCell(...)`;
5. вызвать `Save()`;
6. в новом экземпляре вызвать `OpenAll()`.

## Как устроен `DEObject`

`DEObject` — это одновременно:

- контейнер данных в памяти;
- фасад над сериализацией;
- фасад над десериализацией;
- единственный публичный тип библиотеки. `[confirmed by code]`

Практически это значит, что почти вся интеграция строится вокруг одного объекта.

## Как понять, что данные сохранились корректно

Лучший практический способ:

1. вызвать `Save()`;
2. создать новый `DEObject` с тем же путём;
3. вызвать `OpenAll()`;
4. сравнить полученные данные с исходными.

Если используются матрицы:

5. для каждого маркера `matrix_N` получить массив через `MatrixData(...)` и сравнить размеры/значения.

## Как работает совместимость версий

Текущий новый reader умеет читать старые строковые токены типов и старый разделитель однофайлового режима. `[confirmed by code]`

Обратное неверно: старый код `1.4` не гарантирует чтение файлов, записанных текущей веткой, потому что:

- появились короткие токены типов;
- появился парольный маркер;
- появились матрицы;
- появился маркер `"l"` для серий `int`. `[confirmed by code]`

## Что делать при неясном поведении

- Проверить, тот ли путь хранения используется.
- Проверить, совпадает ли режим `_TOne` при записи и чтении.
- Если был пароль, убедиться, что `Password(...)` вызван до чтения.
- Вызвать `OpenAll()` новым экземпляром и посмотреть, что реально вернулось.
- Проверить вывод в `Console`, потому что часть ошибок библиотека печатает туда, а не пробрасывает наружу. `[confirmed by code]`

## Где границы публичного API

Во внешнем API подтверждён только `GDELib.DEObject`. `[confirmed by code]`

Внутренние типы вроде `DESaver`, `DESMini`, `Yacheyka`, trie-структур и `MatrixIndexStorage`:

- существуют в коде;
- важны для понимания реализации;
- не являются поддерживаемой пользовательской точкой входа.

## Как интерпретировать ограничения формата

- Формат ориентирован на прикладное хранение последовательности ячеек, а не на произвольную бинарную БД.
- Пароль в текущем коде — это проверка доступа к чтению, а не шифрование.
- Файловые ячейки хранятся как zip-вложения с распаковкой в кэш.
- Матрицы используют отдельный компрессионный слой и строковые маркеры `matrix_N`.
- У формата есть эволюция между `1.4` и текущей веткой, поэтому смешивать старые и новые writer/readers без миграции не стоит.

## Почему `OpenAll()` возвращает строки, а не исходные типы

Таков публичный контракт библиотеки. Внутри типы различаются по токенам и внутренним структурам, но наружу `OpenAll()` и `OpenNext()` возвращают строковые представления значений. Для матриц используется отдельный шаг через `MatrixData(...)`. `[confirmed by code]`

## Чем отличаются `OpenNext()` и `NextData()`

- `OpenNext()` читает из файла;
- `NextData()` читает из уже загруженного в память списка.

Если вы уже вызвали `OpenAll()`, обычно выгоднее использовать `NextData()`.

## Можно ли использовать GDELib как защищённое хранилище

Нет, если под "защищённым" понимать шифрование данных на диске.

В текущей реализации:

- считается SHA-256-хэш пароля;
- при чтении он сравнивается с введённым значением;
- полезная нагрузка не шифруется. `[confirmed by code]`

## Почему при работе с файлами нужна папка `cashfile`

Библиотека использует кэш-папку для упаковки и распаковки файловых вложений. Это внутренняя служебная директория, которую `DEObject` старается создать автоматически при инициализации. `[confirmed by code]`

## Почему в документации так много пометок `confirmed by code` и `needs verification`

Потому что в проекте уже есть расхождение между:

- опубликованным пакетом `1.4.0`;
- текущим кодом репозитория;
- веткой разработки `1.5 Preview`.

Чтобы не смешивать подтверждённые факты и предположения, спорные места помечаются явно.
