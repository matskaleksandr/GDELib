# Упаковка данных и размер контейнера

## Что это и зачем нужно

В `GDELib 1.4.0` нет отдельной команды или настройки, которую пользователь включает как «сжатие». Библиотека сама организует хранение данных внутри формата SVE, а для разработчика важен практический результат:

- значения сохраняются в контейнер библиотеки;
- файловые ячейки переносят сам файл, а не только строку с путём;
- контейнер можно хранить либо в двух файлах, либо в одном основном файле.

Проще говоря, этот раздел нужен не для настройки библиотеки, а для понимания, почему итоговый набор файлов на диске выглядит именно так и от чего зависит его размер.

## Что пользователь видит на практике

Для обычных типов (`int`, `double`, `string`, `bool`) библиотека сохраняет значения в контейнер и затем возвращает их в виде строк через `OpenAll()` или `OpenNext(...)`.

Для файловых ячеек поведение другое:

1. приложение передаёт путь к исходному файлу;
2. библиотека сохраняет файл в контейнер;
3. при чтении возвращается путь к восстановленной копии файла.

Это удобно, когда вместе с данными нужно переносить и сам ресурс.

## Как режим хранения влияет на результат

### Двухфайловый режим

По умолчанию библиотека работает с отдельными файлами структуры и данных. Такой вариант обычно выбирают, когда нужна наглядная организация сохранения на диске.

### Однофайловый режим

Если включить `_TOne = true`, библиотека создаёт один основной контейнер. Это полезно, когда проекту удобнее работать с одним пользовательским файлом.

Важно понимать простую вещь: однофайловый режим в первую очередь меняет организацию хранения, а не обещает фиксированный выигрыш по размеру для любых данных.

## От чего зависит размер контейнера

Итоговый размер набора данных в `GDELib 1.4.0` больше всего зависит от:

- количества ячеек;
- длины строковых значений;
- наличия файловых вложений;
- размера самих файлов;
- выбранного режима хранения.

Если в контейнере есть файлы, на размер сильнее влияет уже содержимое этих файлов, а не только количество ячеек.

## Когда что выбирать

| Сценарий | Что обычно удобнее |
| --- | --- |
| Нужна понятная структура на диске | Двухфайловый режим |
| Нужен один основной контейнер для передачи или хранения | Однофайловый режим |
| Нужно сохранить ресурс вместе с данными | Файловая ячейка через `CreateCell("file", path)` |
| Важен реальный размер контейнера в вашем проекте | Проверить оба режима на своих данных |

## Как быстро посмотреть результат на диске

Если нужно понять, что именно создала библиотека после `Save()`, достаточно посмотреть файлы рабочей папки:

```csharp
using System;
using System.IO;
using GDELib;

string storeDir = Path.Combine(Environment.CurrentDirectory, "CompressionView");
Directory.CreateDirectory(storeDir);

var de = new DEObject(storeDir);
de.CreateCell("string", "demo");
de.CreateCell("bool", true);
de.Save();

foreach (string file in Directory.EnumerateFiles(storeDir))
{
    var info = new FileInfo(file);
    Console.WriteLine($"{info.Name}: {info.Length} bytes");
}
```

Такой способ полезен в прикладной разработке: он показывает итоговый результат работы библиотеки без лишнего углубления в устройство формата.

## Практические рекомендации

- Если нужен самый понятный старт, начинайте с двухфайлового режима.
- Если приложение должно оперировать одним основным файлом, используйте однофайловый режим.
- Если добавляете файл через `CreateCell("file", path)`, рассчитывайте место на диске с учётом самого файла.
- Если размер контейнера важен для продукта, сравните оба режима на реальном наборе данных вашего проекта.

## Что важно запомнить

- В `GDELib 1.4.0` нет отдельного пользовательского API для управления сжатием.
- Библиотека сама организует упаковку данных внутри формата SVE.
- Файловые ячейки хранят содержимое файла внутри контейнера.
- Итоговый размер зависит прежде всего от ваших данных и выбранного режима хранения.
