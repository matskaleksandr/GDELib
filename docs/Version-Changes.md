# Изменения в версии 1.4.0

## Что это и зачем нужно

Настоящий документ описывает отличия релиза `GDELib 1.4.0`, которые имеют значение для пользователя библиотеки, интегратора и автора прикладного кода. Основной интерес представляют:

- способ распространения пакета;
- поддерживаемые целевые платформы;
- состояние публичного API;
- практические последствия при переходе с опубликованной версии `0.1.4`.

Сопоставление выполнено по опубликованным NuGet-пакетам `0.1.4` и `1.4.0`, их содержимому и отражению публичного API класса `DEObject`.

## Общая характеристика релиза

Версия `1.4.0` является релизной поставкой библиотеки `GDELib`, ориентированной на использование в прикладных .NET-проектах. Библиотека по-прежнему строится вокруг класса `DEObject` и сохраняет ту же модель работы:

1. создание объекта `DEObject`;
2. добавление ячеек через `CreateCell(...)`;
3. сохранение через `Save()`;
4. чтение через `OpenAll()` или `OpenNext(...)`.

Таким образом, переход к `1.4.0` не требует смены базового сценария работы с библиотекой.

## Состояние публичного API

При сопоставлении опубликованных DLL `0.1.4` и `1.4.0` различий в составе публичных конструкторов и методов `DEObject` не выявлено.

В таблице 2 приведён перечень внешних членов API, доступных в обеих версиях.

Таблица 2 - Публичный API `DEObject`, общий для `0.1.4` и `1.4.0`

| Категория | Элемент |
| --- | --- |
| Конструктор | `DEObject(string _FilePath, bool _TOne = false, string _NameData = "data.sve", string _NameStruct = "struct.sve", string _CashPath = "cashfile")` |
| Методы записи | `CreateCell(string, object)`, `Save()` |

Для пользователя это означает, что код, написанный на уровне публичного API `DEObject`, при переходе с `0.1.4` на `1.4.0` не требует обязательной переработки только из-за смены сигнатур.

## Что изменилось для пользователя

В таблице 3 представлены изменения, которые наиболее заметны в прикладном использовании.

Таблица 3 - Изменения релиза `1.4.0` и их практическое значение

| Изменение | Практическое значение |
| --- | --- |
| Поставка сборок для нескольких целевых платформ | Библиотеку можно подключать не только в проекты `net6.0`, но и в `netstandard2.1` и `net472` |
| Сохранение единой модели работы через `DEObject` | Примеры использования и код интеграции остаются прежними |
| Сохранение однофайлового и двухфайлового режимов | Не требуется менять способ организации контейнеров на уровне прикладной логики |
| Сохранение общей структуры пакета и лицензии | Процедура внедрения в проект остаётся предсказуемой |

Следовательно, релиз `1.4.0` следует рассматривать прежде всего как стабилизированную и более универсальную по платформам поставку уже знакомой модели работы.

## Влияние на миграцию

Для проектов, использующих `GDELib 0.1.4`, практический порядок перехода к `1.4.0` может быть сведён к следующим действиям:

1. обновить пакет до версии `1.4.0`;
2. проверить, какая целевая платформа используется в проекте;
3. выполнить контрольный сценарий записи и чтения через `Save()` и `OpenAll()`;
4. отдельно проверить сценарии с файловыми ячейками, если библиотека используется для хранения файлов.

Такой порядок достаточен для типового прикладного проекта, поскольку различия в публичной API-поверхности между сравниваемыми версиями не обнаружены.

## Вопрос о breaking changes

По результатам сравнения опубликованных пакетов `0.1.4` и `1.4.0` breaking changes в публичном API `DEObject` не выявлены.

При этом следует учитывать, что практическая совместимость должна подтверждаться на собственных данных проекта. Для библиотек хранения это особенно важно, если приложение одновременно использует файловые ячейки, однофайловый режим и ранее созданные контейнеры.

## Краткий вывод

Версия `1.4.0` сохраняет уже существующий внешний способ работы с `GDELib`, но делает поставку библиотеки более универсальной за счёт расширения поддерживаемых целевых платформ. Для конечного пользователя это означает простой переход без смены привычного сценария работы с `DEObject`.
