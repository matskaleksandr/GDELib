# GDELib

GDELib — это C#/.NET-библиотека для сохранения и чтения набора значений в собственном бинарном формате. Библиотека строится вокруг одного публичного класса `DEObject`: через него вы добавляете ячейки с данными, сохраняете их на диск, повторно открываете файл и по необходимости читаете значения по одному.

Библиотека полезна, когда нужен простой прикладной слой над пользовательским форматом хранения, а не полноценная СУБД или общий сериализатор JSON/XML. Типичный сценарий: сохранить небольшой или средний набор игровых, служебных или прикладных данных в компактный файл без написания собственного формата с нуля.

## Что умеет библиотека

- сохранять `int`, `double`, `string`, `bool`, путь к файлу и `int[,]`-матрицы;
- работать в двух режимах:
  - двухфайловый режим: структура и данные хранятся отдельно;
  - однофайловый режим: структура и данные упаковываются в один файл;
- читать весь набор данных сразу или получать значения по одному;
- хранить файлы как zip-вложение во внутреннем формате библиотеки;
- выполнять специализированное сжатие для матриц и для длинных серий целых чисел в однофайловом режиме.

## Когда использовать GDELib

Используйте GDELib, если вам нужен именно прикладной файловый контейнер:

- для игр, утилит и desktop-приложений с собственным форматом сохранений;
- для локальных save/load-сценариев без внешней базы данных;
- когда важнее компактный прикладной API, чем универсальность общего сериализатора;
- когда хотите явно управлять файлами `data/struct`, режимом записи и кэшем вложений.

Лучше не выбирать GDELib как замену БД, потокового архива больших каталогов или защищённого хранилища секретов. Проверка пароля в текущей реализации ограничивает доступ к чтению, но не шифрует сами данные. `[confirmed by code]`

## Подключение как `.dll`

Если вы не используете NuGet, можно подключить библиотеку вручную:

1. Возьмите `GDELib.dll` из соответствующей папки пакета:
   - `lib/net6.0`
   - `lib/netstandard2.1`
   - `lib/net472`
2. Добавьте ссылку на DLL в свой проект.
3. Убедитесь, что целевая платформа вашего приложения совместима с выбранной сборкой.

Практически это удобно в закрытых проектах, офлайн-сборках и там, где пакетный менеджер использовать нельзя.

## Минимальный пример

```csharp
using System;
using System.IO;
using GDELib;

string storeDir = Path.Combine(Environment.CurrentDirectory, "Store");
Directory.CreateDirectory(storeDir);

var writer = new DEObject(storeDir);
writer.CreateCell(42);
writer.CreateCell("Привет, GDELib");
writer.CreateCell(true);
writer.Save();

var reader = new DEObject(storeDir);
string[] values = reader.OpenAll();

Console.WriteLine(string.Join(", ", values));
```

Что происходит в примере:

- создаётся контейнер `DEObject`, связанный с папкой хранения;
- в него добавляются три ячейки;
- `Save()` пишет данные в собственный формат библиотеки;
- новый экземпляр `DEObject` открывает те же файлы и возвращает значения как `string[]`.

## Короткий FAQ

**GDELib шифрует данные?**

Нет. Текущая реализация хранит SHA-256-хэш пароля и сравнивает его при чтении, но не шифрует полезную нагрузку. `[confirmed by code]`

**С чего начинать работу?**

С создания `DEObject`, добавления ячеек через `CreateCell(...)`, вызова `Save()`, а затем чтения через `OpenAll()` или `OpenNext()`.

**Что возвращается для матрицы?**

При чтении API возвращает строковый маркер вида `matrix_3`, а сами данные матрицы нужно получать через `MatrixData("matrix_3")`. `[confirmed by code]`

**Есть ли несколько публичных классов?**

Нет. Во внешнем API библиотеки подтверждён только `DEObject`. Остальные классы реализации объявлены как `internal`. `[confirmed by code]`

## Ограничения и замечания

- `DEObject` — единственная публичная точка входа. Это упрощает API, но делает класс крупным и многофункциональным.
- Ошибки в операциях чтения/записи часто не пробрасываются наружу, а печатаются в `Console`. Для production-сценариев лучше делать round-trip проверку после записи. `[confirmed by code]`
- Однофайловый и двухфайловый режимы не взаимозаменяемы на одном и том же наборе файлов без явного пересохранения данных.
- Текущий код репозитория содержит несколько известных шероховатостей реализации, например особенности `CloneDataDE`, `Dell` и граничные случаи в `NextData`. Документация отдельно отмечает такие места. `[confirmed by code]`
- Формат матриц развивается. Текущий код читает старые строковые токены типов (`integer`, `doubl`, `str`, `booling`, `files`), но старые версии библиотеки не обязаны читать файлы, сохранённые новым кодом. `[confirmed by code]`
